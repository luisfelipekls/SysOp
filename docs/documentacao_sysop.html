1. Visão Geral do Projeto

O SysOp é um simulador educacional de sistema operacional desenvolvido em Java puro para a disciplina de Sistemas Operacionais da PUCRS. O objetivo é demonstrar, de forma didática e executável, os conceitos fundamentais de um SO real: gerenciamento de memória com paginação física, criação e ciclo de vida de processos, execução de programas na CPU simulada e tratamento de interrupções.

O trabalho é dividido em partes incrementais:

ParteNomeO que adicionaStatus
T1-AGerenciador de Memória (GM)Paginação física: alocar/liberar frames, carregar programa em frames não-contíguosConcluído
T1-BGerenciador de Processos (GP)PCB, fila de prontos, criar/remover processo, exec, execAll, CLIConcluído
T1-CEscalonador (Scheduler)Round-Robin com quantum, interrupção de timer, múltiplos processos concorrentesEm andamento
Princípio central
Os programas de usuário usam endereços lógicos contíguos (0, 1, 2, …). O sistema operacional é responsável por mapear esses endereços para frames físicos espalhados pela RAM, usando a tabela de páginas do processo.
Pontos para apresentar nesta seção
  • Contextualizar o problema: por que um SO precisa de gerenciamento de memória e de processos?
  • Mostrar que o projeto simula uma máquina real em miniatura: CPU, RAM, SO, programas.
  • Ressaltar que não há frameworks — tudo construído a partir do zero em Java.
  • Apresentar o diagrama de camadas (hardware → SO → usuário).
Possíveis perguntas do professor
  • Por que simular em Java e não usar C?
    Java oferece coleta de lixo e tipos seguros, reduzindo erros de ponteiro que desviariam o foco do aprendizado. O comportamento do SO é modelado nas classes, não na linguagem.
  • O que diferencia T1-A de T1-B?
    T1-A lida exclusivamente com a gestão de frames físicos (alocação/liberação). T1-B adiciona a abstração de processo: PCB, fila de prontos, ciclo de vida e execução da CPU em contexto de processo.

3. Camada de Hardware Simulado (Sistema.java)

Toda a simulação de hardware reside em src/system/Sistema.java. A classe usa inner classes para agrupar logicamente os componentes de hardware, mantendo acesso privilegiado entre eles sem expor detalhes ao SO.

3.1 Memória (Memory)

A RAM é modelada como um vetor de Word. O tamanho padrão é 1.024 palavras. Cada posição é inicializada com o opcode ___ (nulo), indicando memória não usada.

public class Memory {
    public Word[] pos; // pos[i] é a posição i da memória

    public Memory(int size) {
        pos = new Word[size];
        for (int i = 0; i < pos.length; i++) {
            pos[i] = new Word(Opcode.___, -1, -1, -1); // inicializa como vazia
        }
    }
}
Detalhe importante
A memória não distingue código de dados — toda posição é um Word. O opcode DATA sinaliza que aquela posição contém um dado (não deve ser executada como instrução). Isso reflete a arquitetura de Von Neumann.

3.2 Palavra (Word)

Cada célula de memória é uma instrução de 4 campos. Essa estrutura unificada permite que a CPU leia tanto instruções quanto dados do mesmo array.

public static class Word {
    public Opcode opc; // operação a realizar
    public int ra;     // registrador destino (ou primeiro operando)
    public int rb;     // registrador fonte (ou segundo operando)
    public int p;      // parâmetro imediato (endereço, constante ou dado)

    public Word(Opcode _opc, int _ra, int _rb, int _p) {
        opc = _opc; ra = _ra; rb = _rb; p = _p;
    }
}

Exemplo: new Word(Opcode.LDI, 0, -1, 7) representa a instrução r[0] ← 7. Campos não usados recebem -1.

3.3 CPU e Ciclo de Instrução

A CPU implementa o ciclo clássico Fetch → Decode → Execute → Verificar Interrupção. Ela possui 10 registradores de uso geral, PC, IR e uma flag de interrupção.

public class CPU {
    private int pc;           // Program Counter (endereço lógico atual)
    private Word ir;          // Instruction Register
    private int[] reg;        // 10 registradores de uso geral (reg[8] e reg[9] para I/O)
    private Interrupts irpt;  // sinaliza interrupção durante a execução

    private int pageSize;     // tamanho de página (para tradução de endereços)
    private PCB currentProcess; // processo em execução (para acessar tabela de páginas)

    public void run(PCB runningProcess, boolean isExecAll) {
        this.currentProcess = runningProcess;
        System.arraycopy(runningProcess.savedRegisters, 0, reg, 0, reg.length); // restaura contexto
        Integer cicleLimit = 5; // quantum: máximo de instruções por rodada
        Integer cicle = 0;
        cpuStop = false;

        while (!cpuStop) {
            // --- Preempção por quantum (apenas em execAll) ---
            if (isExecAll && cicle >= cicleLimit) {
                sysCall.stop();
                cpuStop = true;
                System.arraycopy(reg, 0, runningProcess.savedRegisters, 0, reg.length); // salva contexto
                processManager.pcbReadyList.add(runningProcess); // retorna à fila
                break;
            }

            // --- FASE 1: FETCH ---
            if (legal(pc)) {
                ir = m[translate(pc)];    // busca instrução: traduz PC lógico → físico
                runningProcess.processPc = pc; // salva PC no PCB

                // --- FASE 2/3: DECODE + EXECUTE ---
                switch (ir.opc) {
                    case LDI: reg[ir.ra] = ir.p; pc++; break;
                    case ADD:  reg[ir.ra] = reg[ir.ra] + reg[ir.rb]; testOverflow(reg[ir.ra]); pc++; break;
                    case STD:
                        if (legal(ir.p)) { m[translate(ir.p)].p = reg[ir.ra]; pc++; }
                        break;
                    case STOP: sysCall.stop(); cpuStop = true; break;
                    // ... demais opcodes
                }
            }

            // --- FASE 4: VERIFICAR INTERRUPÇÃO ---
            if (irpt != Interrupts.noInterrupt) {
                ih.handle(irpt);
                cpuStop = true;
            }
            cicle++;
        }

        if (!interuption) runningProcess.status = ProcessStatus.FINISHED;
    }
}
Quantum de 5 instruções
O limite de 5 ciclos só é aplicado quando isExecAll = true. No modo exec (processo único), a CPU roda até o processo terminar, encontrar erro ou chamar STOP.

3.5 Tratamento de Interrupções

O mecanismo de interrupção é sinalizado pela flag irpt dentro do ciclo da CPU. Ao fim de cada instrução, a CPU verifica se uma interrupção foi gerada e desvia para o InterruptHandling do SO.

public enum Interrupts {
    noInterrupt,         // nenhuma interrupção pendente
    intEnderecoInvalido, // acesso fora do espaço do processo
    intInstrucaoInvalida,// opcode DATA sendo executado, ou opcode desconhecido
    intOverflow          // resultado aritmético fora da faixa [-32767, 32767]
}

// Verificação de endereço lógico válido:
private boolean legal(int e) {
    if (e >= 0 && currentProcess != null && e < currentProcess.pages.length * pageSize) {
        return true;
    } else {
        irpt = Interrupts.intEnderecoInvalido;
        return false;
    }
}
Pontos para apresentar nesta seção
  • Demonstrar ao vivo o ciclo fetch-execute com o modo traceOn ativado — a CPU imprime cada instrução e o estado dos registradores.
  • Explicar por que legal() usa o espaço lógico do processo e não o endereço físico diretamente.
  • Mostrar a tabela de opcodes e ressaltar o prefixo K (imediato) vs M (indireto via memória).
  • Falar sobre overflow aritmético: a CPU simula um inteiro de 16 bits (-32767 a 32767).
Possíveis perguntas do professor
  • O que acontece quando a CPU acessa um endereço fora dos limites do processo?
    A função legal() sinaliza intEnderecoInvalido. Ao fim do ciclo, o InterruptHandling é acionado e a execução do processo é interrompida.
  • Por que o overflow é tratado como interrupção e não como exceção Java?
    Para simular fielmente o comportamento de hardware. Em processadores reais, overflow dispara uma interrupção que o SO pode tratar — ignorar, encerrar o processo ou sinalizar ao programa.
  • Como funciona o SYSCALL de I/O?
    O programa carrega em reg[8] o tipo (1=leitura, 2=escrita) e em reg[9] o endereço lógico do dado. A instrução SYSCALL desvia para SysCallHandling.handle(), que imprime ou lê o valor.
  • Por que o PC é mantido como endereço lógico, não físico?
    O processo "pensa" em endereços lógicos. O PC representa a posição no espaço de endereçamento virtual do processo. A tradução para físico acontece no momento do fetch, dentro de translate(pc).

4. Gerenciador de Memória Física — T1-A

O pacote memoria implementa a paginação física. A memória é dividida em frames de tamanho fixo (padrão: 16 palavras). Processos recebem um conjunto de frames — não necessariamente contíguos — que formam suas páginas.

4.1 Frame (Frame.java)

Um frame representa um bloco físico de memória. É a unidade básica de alocação.

public class Frame {
    public static int indexCounter = 0; // contador global de frames
    public int index;       // número único do frame (0 a 63)
    public int start;       // endereço físico de início (ex: frame 2 → start=32)
    public int end;         // endereço físico de fim   (ex: frame 2 → end=47)
    public boolean isOccupied; // true se alocado a algum processo

    public Frame(int start, int end) {
        this.start = start;
        this.end = end;
        this.index = indexCounter++;
        // isOccupied inicializa false por padrão
    }
}

Fórmula: frame de índice f ocupa posições físicas [f × pageSize, (f+1) × pageSize − 1].

4.2 Página (Page.java)

Uma Page associa uma página lógica do processo a um frame físico. É através da tabela de páginas (array PCB.pages[]) que a CPU realiza a tradução de endereços.

public class Page {
    public Frame frame; // referência ao frame físico correspondente
    Word[] words;       // cache das words (não essencial para execução)

    public Page(Frame frame, int size) {
        this.frame = frame;
        this.words = new Word[size];
    }
}

4.3 Gerenciador de Memória (MemoryManager.java)

O MemoryManager mantém um vetor frameControl[] com todos os frames da memória. Ele é a única entidade autorizada a marcar frames como ocupados ou livres.

public class MemoryManager {
    private int pageSize;
    private Memory memory;
    private Frame[] frameControl; // um Frame para cada bloco físico de pageSize words

    public MemoryManager(Memory memory, int pageSize) {
        this.memory = memory;
        this.pageSize = pageSize;
        this.frameControl = new Frame[memory.pos.length / pageSize]; // 1024/16 = 64
        initializeFrameControl();
    }

    private void initializeFrameControl() {
        for (int i = 0; i < frameControl.length; i++) {
            // Frame i ocupa [i*pageSize, (i+1)*pageSize - 1]
            frameControl[i] = new Frame(i * pageSize, ((i + 1) * pageSize) - 1);
        }
    }
}

Alocação: allocate(nWords)

Calcula quantos frames são necessários para nWords palavras e retorna um array de Page com os frames alocados. Usa divisão com teto (ceiling division) para garantir que sobras caibam em um frame extra.

public Page[] allocate(int wordsSize) {
    int numPages = (wordsSize + pageSize - 1) / pageSize; // teto: ex. 20 words / 16 = 2 frames

    Frame[] available = getAvailableFrames(); // filtra frames com isOccupied == false
    if (numPages > available.length) {
        return null; // sem memória disponível
    }

    Page[] pages = new Page[numPages];
    for (int i = 0; i < numPages; i++) {
        available[i].isOccupied = true; // marca como ocupado
        pages[i] = new Page(available[i], pageSize);
    }
    return pages;
}

public Frame[] getAvailableFrames() {
    return Arrays.stream(frameControl)
            .filter(frame -> !frame.isOccupied)
            .toArray(Frame[]::new);
}

Liberação: deallocate(pages[])

Libera todos os frames de um processo, zerando a memória física e marcando os frames como disponíveis.

public void deallocate(Page[] pages) {
    for (Page page : pages) {
        page.words = null;
        page.frame.isOccupied = false;
        // limpa a memória física do frame
        for (int i = page.frame.start; i <= page.frame.end; i++) {
            memory.pos[i] = new Word(Opcode.___, -1, -1, -1);
        }
    }
}
Estratégia de alocação: First-Fit implícito
O getAvailableFrames() retorna frames livres em ordem de índice. O allocate() usa sempre os primeiros da lista — comportamento equivalente ao First-Fit. Em cenários com muita fragmentação externa, isso poderia ser substituído por Best-Fit.
Pontos para apresentar nesta seção
  • Demonstrar visualmente a fragmentação: criar dois processos, remover o primeiro, criar um terceiro maior — os frames do primeiro ficam disponíveis para reutilização.
  • Mostrar o comando dumpM que exibe a quantidade de frames livres e ocupados.
  • Explicar por que usamos paginação em vez de segmentação ou alocação contígua.
  • Calcular ao vivo: programa com 20 instruções → ceil(20/16) = 2 frames necessários.
Possíveis perguntas do professor
  • Por que a divisão usa (wordsSize + pageSize - 1) / pageSize e não simplesmente wordsSize / pageSize?
    A divisão inteira trunca para baixo. Se o programa tem 17 words e o frame tem 16, 17/16 = 1 (errado — faltaria 1 word). A fórmula de teto garante que o resto seja acomodado: (17+15)/16 = 2 frames.
  • O que acontece se não há frames livres suficientes?
    allocate() retorna null. O ProcessManager detecta isso, imprime mensagem de erro e a criação do processo falha sem corromper o estado do sistema.
  • Por que os frames são zerados na liberação?
    Para evitar vazamento de informação entre processos — um processo novo não deve ver dados do processo anterior. Em sistemas reais, isso é obrigatório por segurança.
  • Qual a diferença entre fragmentação interna e externa nesse modelo?
    Fragmentação interna: o último frame de um processo pode estar parcialmente vazio (ex: 3 words usadas de 16). Fragmentação externa não existe em paginação pura — qualquer conjunto de frames livres pode ser usado.

5. Gerenciador de Processos — T1-B

5.1 Process Control Block (PCB.java)

O PCB é a representação interna de um processo dentro do SO. Contém tudo que o sistema precisa saber sobre o processo para pausar e retomar sua execução.

public class PCB {
    private static int idCounter = 0;   // PID auto-incremento global

    public Integer id;                  // PID único do processo
    public ProcessStatus status;        // estado atual (CREATED, READY, EXECUTING…)
    public Frame[] frames;              // frames físicos alocados para este processo
    public Page[] pages;                // tabela de páginas (mapeamento lógico → físico)
    public Integer processPc;           // PC salvo (endereço lógico para retomar execução)
    public int[] savedRegisters = new int[10]; // contexto dos 10 registradores (context switch)
    private String programName;         // nome do programa (para exibição)

    PCB() {
        id = idCounter++;
        status = ProcessStatus.CREATED;
        programName = "unknown";
    }

    public void setFrames(Frame[] programFrames) {
        frames = programFrames;
        processPc = 0; // PC inicia em 0 (endereço lógico 0 = início do programa)
    }

    public void setPages(Page[] programPages) {
        pages = programPages;
    }
}
Context Switch via savedRegisters
Quando um processo é preemptado (quantum esgotado em execAll), a CPU copia reg[] para pcb.savedRegisters[] antes de liberar a CPU. Quando o processo retoma, os registradores são restaurados de savedRegisters[] antes de iniciar o ciclo.

5.2 Estados do Processo (ProcessStatus.java)

public enum ProcessStatus {
    CREATED,    // PCB criado, ainda não entrou na fila de prontos
    READY,      // na fila de prontos, aguardando CPU
    EXECUTING,  // usando a CPU neste momento
    PAUSE,      // interrompido por preempção (quantum), voltará à fila
    FINISHED    // terminou (STOP ou erro fatal) — será removido
}
criar processo ─────────────────────────────► CREATED │ add à fila │ de prontos ▼ READY ◄────────────┐ │ │ escalonador │ │ preempção seleciona ▼ │ (quantum) EXECUTING ──────────►┘ │ STOP/erro │ ▼ FINISHED

5.3 Gerenciador de Processos (ProcessManager.java)

O ProcessManager é o coração do T1-B. Ele orquestra todos os aspectos do ciclo de vida de processos.

Criação de processo: createProcess(name)

public boolean createProcess(String programName) {
    Word[] programImage = programs.retrieveProgram(programName); // busca programa em "disco"
    if (programImage == null) {
        System.out.println("[PM] Erro: Programa '" + programName + "' não encontrado.");
        return false;
    }
    return createProcessFromImage(programImage, programName);
}

private boolean createProcessFromImage(Word[] programImage, String programName) {
    // 1. Alocar páginas (e frames) no MemoryManager
    Page[] programPages = memoryManager.allocate(programImage.length);
    if (programPages == null) return false; // sem memória

    // 2. Extrair os frames das páginas
    Frame[] frames = getFramesPromPages(programPages);

    // 3. Criar e configurar o PCB
    PCB currentProgramPCB = new PCB();
    currentProgramPCB.setProgramName(programName);
    currentProgramPCB.setFrames(frames);   // seta frames e inicializa processPc = 0
    currentProgramPCB.setPages(programPages);

    // 4. Copiar o código para os frames físicos
    loadProgramInMemory(currentProgramPCB, programImage);

    // 5. Adicionar à fila de prontos
    pcbReadyList.add(currentProgramPCB);
    System.out.println("[PM] Processo criado: PID=" + currentProgramPCB.id);
    return true;
}

Carregamento na memória: loadProgramInMemory()

Copia as words do programa para os frames físicos em ordem, respeitando os limites de cada frame. Isso transforma o programa (que usa endereços lógicos contíguos) em código espalhado pela RAM física.

private void loadProgramInMemory(PCB currentProgramPCB, Word[] programImage) {
    int frameSize = currentProgramPCB.frames[0].end - currentProgramPCB.frames[0].start + 1; // 16
    int countWords = 0;

    for (int i = 0; i < currentProgramPCB.frames.length; i++) {    // para cada frame
        for (int j = 0; j < frameSize; j++) {                       // para cada posição no frame
            if (countWords < programImage.length) {
                memory.pos[currentProgramPCB.frames[i].start + j] = programImage[countWords];
                countWords++;
            }
            // posições restantes no último frame ficam com Word(___, -1, -1, -1)
        }
    }
}

Execução: executeProcess(pid, isExecAll)

public void executeProcess(int pid, boolean isExecAll) {
    PCB pcb = getProcess(pid);
    if (pcb == null) { System.out.println("[PM] Processo não encontrado."); return; }

    running = pcb;
    pcb.status = ProcessStatus.EXECUTING;

    int startAddress = pcb.processPc; // retoma de onde parou (0 na primeira vez)
    cpu.setContext(startAddress);     // seta PC da CPU (endereço lógico)

    pcbReadyList.remove(pcb);        // remove temporariamente da fila durante execução
    cpu.run(running, isExecAll);     // executa (com ou sem quantum)

    if (pcb.status == ProcessStatus.FINISHED) {
        memoryManager.deallocate(running.pages); // libera memória automaticamente
        System.out.println("[PM] Processo PID=" + pid + " finalizado.");
        running = null;
    } else {
        System.out.println("[PM] Processo PID=" + pid + " interrompido.");
        // processo voltou para pcbReadyList dentro da CPU (preempção)
    }
}

// execAll: roda cada processo na fila até todos terminarem
public void execAll() {
    while (!pcbReadyList.isEmpty()) {
        executeProcess(pcbReadyList.get(0).id, true);
    }
}

Remoção: dealocateProcess(id)

public void dealocateProcess(int id) {
    PCB currentPCB = pcbReadyList.stream()
            .filter(a -> a.id == id).findFirst().get(); // busca o processo

    memoryManager.deallocate(currentPCB.pages); // libera frames
    pcbReadyList.remove(currentPCB);             // remove da fila
    dealocateProcessInMemory(currentPCB);        // zera a memória física
}

private void dealocateProcessInMemory(PCB currentPCB) {
    int frameSize = currentPCB.frames[0].end - currentPCB.frames[0].start + 1;
    for (int i = 0; i < currentPCB.frames.length; i++) {
        for (int j = 0; j < frameSize; j++) {
            memory.pos[currentPCB.frames[i].start + j] = new Word(Opcode.___, -1, -1, -1);
        }
    }
}
Nota de implementação
A memória é limpa tanto em MemoryManager.deallocate() quanto em dealocateProcessInMemory(). Há uma sobreposição — na prática, uma das duas é redundante. Isso não causa erro, apenas limpeza dupla.

Possíveis perguntas do professor
  • O que é um PCB e quais informações mínimas ele precisa conter?
    PCB = Process Control Block. É a estrutura de dados que o SO usa para representar um processo. Mínimo necessário: PID, estado, PC salvo, registradores salvos, tabela de páginas (ou ponteiro para ela) e informações de scheduling.
  • Como o processo é "pausado" e "retomado" corretamente?
    Pausa: a CPU salva reg[] em pcb.savedRegisters[] e o PC lógico em pcb.processPc. Retomada: cpu.setContext(pcb.processPc) restaura o PC, e System.arraycopy(savedRegisters, reg) restaura os registradores.
  • O que acontece se tentarmos fazer rm de um processo em execução?
    A implementação atual busca o processo apenas na pcbReadyList. Um processo em execução está em running, fora da lista, então findFirst().get() lançaria NoSuchElementException. Uma melhoria seria verificar running também.
  • Por que o processo é removido da fila de prontos antes de executar?
    Para evitar que o mesmo processo seja selecionado novamente durante a execução (especialmente em execAll). Ele só retorna à fila se for preemptado — caso contrário, termina e é descartado.

6. Tradução de Endereços Lógicos em Físicos

Este é o núcleo conceitual do trabalho — a ponte entre o que o programa enxerga (endereços lógicos contíguos) e o que a RAM física possui (frames espalhados).

6.1 Fórmula de Tradução

Endereço lógico → número da página + offset → endereço físico endereçoLógico = 34 pageSize = 16 página = 34 / 16 = 2 (índice da página na tabela) offset = 34 % 16 = 2 (posição dentro do frame) frame físico da página 2 = pages[2].frame.start = 48 (ex: processo em frames 0, 1, 2) endereçoFísico = 48 + 2 = 50

6.2 Implementação na CPU

// Na classe CPU (Sistema.java):
public int translate(int logicalAddr) {
    int page   = logicalAddr / pageSize;           // qual página lógica
    int offset = logicalAddr % pageSize;           // posição dentro da página
    return currentProcess.pages[page].frame.start + offset; // endereço físico
}

// Uso no fetch da instrução:
ir = m[translate(pc)];  // busca instrução no endereço físico correspondente ao PC lógico

// Uso em LDD (load direto):
case LDD: // Rd <- [A] — A é endereço lógico
    if (legal(ir.p)) {
        reg[ir.ra] = m[translate(ir.p)].p; // traduz antes de acessar a memória
        pc++;
    }
    break;

6.3 Verificação de Acesso (legal())

Antes de qualquer tradução, a CPU verifica se o endereço lógico está dentro dos limites do processo. Isso previne que um processo acesse memória de outro processo.

private boolean legal(int e) {
    // e é um endereço lógico
    // o espaço total do processo = número de páginas × tamanho da página
    if (e >= 0 && currentProcess != null && e < currentProcess.pages.length * pageSize) {
        return true;
    } else {
        irpt = Interrupts.intEnderecoInvalido; // dispara interrupção de segmentation fault
        return false;
    }
}

Possíveis perguntas do professor
  • O que é a fragmentação interna nesse modelo e onde ela ocorre?
    No último frame de um processo: se o programa tem 17 words e o frame tem 16, o segundo frame tem apenas 1 word usada e 15 desperdiçadas. Esse desperdício = fragmentação interna. Frames intermediários não têm fragmentação.
  • Como a paginação garante proteção entre processos?
    Cada processo tem sua própria tabela de páginas (PCB.pages[]). A CPU usa apenas essa tabela para traduzir endereços. Um processo com páginas mapeadas para frames F0 e F1 nunca "enxerga" frames F2+ de outro processo, pois seu espaço lógico acaba no limite de suas páginas — qualquer acesso além disso é barrado por legal().
  • Qual seria a complexidade de adicionar memória virtual (swap) a esse sistema?
    Precisaríamos adicionar: bit de validade na tabela de páginas, rotina de page fault, área de swap em disco (simulado), e algoritmo de substituição (LRU, Clock…). O projeto já inclui GerenciadorMemoriaVirtual.java como extensão opcional com algoritmo Second Chance.

9. Escalonamento — T1-C (Visão Futura)

O T1-C adiciona o escalonador Round-Robin completo. A base já está preparada no código atual:

  • Quantum de 5 instruções: implementado no loop da CPU (cicleLimit = 5)
  • Salvamento de contexto: savedRegisters[] e processPc já são salvos na preempção
  • Retorno à fila: processo preemptado é re-inserido em pcbReadyList
  • execAll: já implementa Round-Robin sequencial de forma simplificada

O que T1-C precisa adicionar:

  1. Interrupção de timer real: disparada automaticamente após N ciclos, não apenas quando isExecAll=true
  2. Thread separada para CPU: CPU em thread própria, bloqueada em semáforo, desbloqueada pelo escalonador
  3. Thread do escalonador: acordada pela interrupção de timer, seleciona próximo processo
  4. Fila de bloqueados: para processos aguardando I/O

Possíveis perguntas do professor (T1-C)
  • O que é Round-Robin e por que usar quantum?
    Round-Robin é um algoritmo de escalonamento preemptivo que aloca uma fatia de tempo (quantum) igual para cada processo na fila circular. Quantum garante que nenhum processo monopolize a CPU — essencial para sistemas interativos.
  • Qual o impacto do tamanho do quantum no desempenho?
    Quantum muito pequeno: muitos context switches, overhead alto. Quantum muito grande: comportamento similar a FIFO, menor responsividade. O valor ótimo depende do tempo médio de burst de CPU dos processos.
  • Por que usar semáforos entre as threads?
    Para sincronização: a Thread CPU não pode executar sem que o escalonador a autorize, e o escalonador não pode selecionar um processo enquanto a CPU está executando. Semáforos garantem exclusão mútua e coordenação de ordem.
